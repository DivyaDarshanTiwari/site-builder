# Non-Functional Requirements

## 4.1 Performance Requirements
- **Real-time System Considerations**: As the core functionality revolves around code generation and revisions via an AI pipeline (OpenAI models), processing time may vary. Generation via the `google/gemini-2.5-flash` model must resolve and transmit the compiled HTML view back to the interface reliably within approximately 5-15 seconds under normal load.
- **Client Render Performance**: The client must load initial scripts and render the base React components in under 2 seconds. Given large generated HTML blocks, React updates modifying the `current_code` state for preview must occur minimally impacting the main thread, completing essentially instantaneously (<100ms) to ensure smooth user interactions.
- **Database Query Performance**: Real-time retrieval of user conversation histories and prior versions using Prisma should have indexing strategies in PostgreSQL, keeping query execution times predictably low (sub 500ms for routine lookups), even as a single `WebsiteProject` accumulates a high number of associated interaction snapshots over time.

## 4.2 Security Requirements
- **Session Authentication & Validation**: System protection and data protection are managed by the `better-auth` library using token verifications securely passed via HTTP headers. Unauthenticated clients must be met with HTTP `401 Unauthorized` via middleware filters before logic execution can manipulate or query Prisma schemas, including user profiles and private content.
- **Data Protection**: All communications strictly mandate the use of HTTPS/TLS configurations to encrypt the AI generated responses, source code iterations, and user credentials. Storing `current_code` in PostgreSQL mitigates XSS injections during transmission by safely encoding generated HTML until deliberate rendering inside secure iframes is executed on the frontend client.
- **User Privacy**: Access to `WebsiteProject`, `Conversation`, and `Version` entities is isolated based on `userId` keys generated during authentication to guarantee that project revisions are visible only to the authorized creator unless the `isPublished` Boolean specifically marks a project viewable to the public.

## 4.3 Software Quality Attributes
- **Availability & Reliability**: The decoupling of the Vite React Frontend from the Express Node.js Backend guarantees robust, partial system availability. If frontend build networks degrade, backend API capabilities are independently verifiable via POST routes.
- **Maintainability & Reusability**: The implementation employs TypeScript typing in both frontend properties and backend handlers (like Express routes), and Prisma strongly types the database layer. This end-to-end type safety greatly improves structural maintainability and module reusability for developers.
- **Portability & Interoperability**: The client connects purely over a standardized JSON REST API structure rendering it highly portable across browser ecosystems and compatible with modern HTTP libraries (e.g., Axios). Code dependencies manage the application in package.json files allowing it to be compiled and run interchangeably across varying OS platforms.
- **Flexibility & Adaptability**: By isolating the prompt enhancement and code generation logic sequentially using an AI pipeline in controller classes, it will be facile to swap in differing AI models or extend prompt parameters without fundamentally breaking the project data architecture.
- **Usability**: The frontend React app features intuitive layout construction, implementing instant code-to-preview translations alongside dynamic rollback capabilities to prioritize the user experience when revising complex UI changes.
- **Correctness & Robustness**: Explicit error boundaries globally trap asynchronous anomalies within Express handling routines resulting in standard `500` reporting preventing memory leaks or uncaught server crashing, fulfilling a robust backend design.
- **Testability**: Segmented routing files (e.g., `userRoutes.js`, `projectRoutes.js`) encapsulate logical domains allowing granular unit testing using tools to mock database and HTTP boundaries explicitly for predictable validation runs prior to deployment configurations.
